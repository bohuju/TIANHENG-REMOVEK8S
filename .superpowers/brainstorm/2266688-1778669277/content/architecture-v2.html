2. 经验建议机制 — 节点主动提议查询 GBrain

不再是被动等待调用，而是主动建议：

plan 节点进入时

💡 MemoryAdapter 主动提示：

"检测到目标仓库 libxml2-v2.9.14，GBrain 中有 3 条相关记录：
  • 上次 fuzz 覆盖率 62%，parser 模块是瓶颈
  • 曾发现 2 个真实漏洞（CVE-2024-xxx, CVE-2025-xxx）
  • 有效 harness 模式：dict-based + ASAN
是否参考这些经验？"

crash-triage 节点进入时

💡 MemoryAdapter 主动提示：

"当前崩溃 SIGSEGV in xmlParseElement，GBrain 中有 2 次相似崩溃记录：
  • 1 次确认为真实漏洞（use-after-free）
  • 1 次确认为 harness 自身 bug
建议对比崩溃特征：调用栈深度、内存状态"

crash-analysis 节点进入时

💡 MemoryAdapter 主动提示：

"相似漏洞判定记录：libxslt 中也出现过类似的解析器 UAF，
根因是 xmlFreeNode 后未置空指针。建议检查当前崩溃是否同类问题。"

3. memory-summarize 节点的职责

子步骤 输入 对 GBrain 的操作
1. 聚合本会话所有阶段结果 plan 决策、harness 生成记录、构建结果、覆盖率时间线、crash 分诊/分析结论 put_page: type=fuzz/session
记录会话时间线
2. 提炼有效策略模式 哪些 seed family 产出高、哪些 harness 结构编译通过率高 put_page: type=fuzz/strategy
add_link → target repo
3. 归档崩溃结论 每个 crash 的最终判定（真实漏洞/误报/未确定） put_page: type=fuzz/crash
add_link → session page
4. 更新目标仓库画像 最新覆盖率、已发现攻击面、推荐策略 get_page → 更新 compiled_truth
add_timeline_entry
5. 建立跨页面关系 session → target repo → crash pages → strategy pages add_link batch
形成知识图谱

4. MemoryAdapter 的主动建议逻辑

每个需要经验的节点进入时，MemoryAdapter 执行：

  1. 查询：根据当前节点上下文（目标仓库、崩溃签名、阶段类型）构造查询，调 GBrain hybrid search
  2. 判断相关性：如果返回结果的 score 超过阈值 → 生成建议提示
  3. 呈现建议：将历史经验以结构化建议形式呈现给节点，节点决定是否采纳
  4. 记录采纳/忽略：无论节点是否采纳，建议本身作为 timeline entry 写入，形成决策轨迹

有建议的节点

  • plan — 查询同类仓库 fuzz 策略
  • crash-triage — 查询相似崩溃分类
  • crash-analysis — 查询相似漏洞根因
  • coverage-analysis — 查询历史覆盖率改进路径

无建议的节点

  • init / synthesize / build / run — 执行确定性操作，不查 GBrain
  • improve-harness / fix-harness — 局部的代码修改，不需要经验
