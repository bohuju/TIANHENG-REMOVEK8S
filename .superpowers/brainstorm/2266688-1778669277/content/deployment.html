从开发到生产，分阶段交付

1. 部署拓扑

开发环境：stdio 子进程模式
┌──────────────────────────────────────────┐
│         Sherpa Pod / 开发机               │
│                                          │
│  ┌─────────┐    MCP stdio    ┌─────────┐ │
│  │ Sherpa   │◄──────────────►│ GBrain  │ │
│  │ workflow │  JSON-RPC      │ serve   │ │
│  │ graph    │  (子进程)       │         │ │
│  └────┬─────┘                └────┬────┘ │
│       │                           │      │
│  ┌────┴─────┐              ┌──────┴────┐ │
│  │ Sherpa   │              │  GBrain   │ │
│  │ Postgres │              │  PGLite   │ │
│  │          │              │  (嵌入式)  │ │
│  └──────────┘              └───────────┘ │
└──────────────────────────────────────────┘

  MemoryAdapter.__init__: 启动 gbrain serve 子进程
  GBrain 数据库: PGLite（零配置，数据在 ~/.gbrain/）
生产环境：K8s Sidecar 模式
┌──────────────────────────────────────────────┐
│              Kubernetes Pod                   │
│                                              │
│  ┌──────────────┐    MCP stdio   ┌─────────┐ │
│  │ Sherpa Job   │◄──────────────►│ GBrain  │ │
│  │ Container    │   localhost    │ Sidecar │ │
│  │ (Python)     │                │ (Bun)   │ │
│  └──────┬───────┘                └────┬────┘ │
│         │                             │      │
└─────────┼─────────────────────────────┼──────┘
          │                             │
    ┌─────┴──────┐             ┌────────┴──────┐
    │  Sherpa    │             │  GBrain       │
    │  Postgres  │             │  Postgres     │
    │  (控制面DB) │             │  + pgvector   │
    └────────────┘             │  (记忆DB)     │
                               └───────────────┘

  GBrain 从 PGLite 迁移到 Postgres + pgvector
  共享 Pod 内的 localhost 通信，零网络延迟
  两张数据库独立，可分别备份/扩缩容

2. 分阶段实现路线

Phase 1 — 核心读写通路（1-2 周）
  • MemoryAdapter 基础实现（MCP client + 核心方法）
  • crash-triage / crash-analysis 节点的实时写入
  • plan 节点的查询（get_target_profile）
  • 开发环境 PGLite，stdio 子进程模式
  • 验证方式：跑完一次完整 fuzz 会话，确认 GBrain 中有 session + crash 记录
Phase 2 — 经验总结节点 + 批量写入（1 周）
  • 实现 memory-summarize 工作流节点
  • batch_write 攒批机制 + 重试逻辑
  • coverage-analysis / build / run 阶段的批量写入
  • session 级 page 间 link 关系建立
  • 验证方式：多次 fuzz 后，GBrain 图谱中能遍历 session→crash→target 链路
Phase 3 — 主动建议机制（1 周）
  • get_suggestions 实现（组合查询 + 结构化建议生成）
  • 建议采纳/忽略的 timeline 记录
  • 搜索阈值调优（避免无效建议）
  • GitNexus code_query 集成（代码级搜索）
  • 验证方式：第二次 fuzz 同一仓库时，plan 节点收到策略建议且建议有效
Phase 4 — 生产化 + 进阶特性（持续）
  • K8s sidecar 部署 + Postgres 迁移
  • 跨仓库知识迁移（相似仓库的策略自动推荐）
  • 策略效果自动评估（success_rate 动态更新）
  • GBrain doctor 定期维护集成

4. 设计总结

决策项 结论
集成模式A. Memory Sidecar — MCP 独立服务
读写策略关键阶段同步实时写，其余会话结束批量写
经验提炼独立 memory-summarize 节点，会话后运行
