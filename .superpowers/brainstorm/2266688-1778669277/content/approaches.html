GBrain × Sherpa 集成：三种架构方案

核心目标：让 Sherpa 的每一次 fuzz 决策、crash 分析、策略经验都沉淀为 GBrain 中的长期记忆，后续任务可检索和复用

A

Memory Sidecar — MCP 独立服务

GBrain 作为独立 MCP Server 运行，Sherpa 工作流节点通过 MCP 协议远程调用 search / query / put_page 等工具。两张数据库独立。

优点

  • 两个系统完全解耦，独立升级
  • GBrain 可服务多个 Sherpa 实例
  • 符合 GBrain 现有 MCP 架构

缺点

  • 每次读写有 MCP 协议开销
  • 需要维护两套数据库
  • 网络依赖，MCP Server 挂了记忆就断
B

Embedded Engine — 引擎内嵌

Sherpa 直接引入 GBrain 的 operations.ts / engine 层作为库，共享同一 Postgres 实例。函数级调用，零网络延迟。

优点

  • 零网络延迟，直接函数调用
  • 单数据库，运维简单
  • 可深度定制 GBrain 行为

缺点

  • 紧耦合，依赖 GBrain 内部 API
  • 升级 GBrain 可能影响 Sherpa
  • GBrain 用 Bun 运行时，Sherpa 用 Python，跨语言调用复杂
C

Knowledge Bridge — 适配器 + 缓存层（推荐）

在 Sherpa 侧建一个轻量 MemoryAdapter，负责：将 fuzz 概念翻译为 GBrain page 结构；本地缓存热点查询；批量异步写入。底层通过 MCP 调用 GBrain。

优点

  • 兼顾解耦和性能（缓存 + 批量写入）
  • 领域翻译层让两边各自演进
  • 写入失败可重试，不影响 fuzz 主流程

缺点

  • 多了一层代码要维护
  • 缓存一致性需要处理

方案对比

维度 A. Sidecar B. Embedded C. Bridge
耦合度 低 高 低
延迟 中 低 低（缓存命中）
运维复杂度 中 低 中
跨语言兼容 好（协议标准） 差（TS↔Python） 好
容错性 中 差（强依赖） 好（重试+降级）
